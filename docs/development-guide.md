# 開発ガイド

## 実装方針

- **学習目的**: コードを理解しながら小さく進める。一度に大量実装しない
- **Issue駆動**: 各フェーズをGitHub Issueで管理。1 Issue = 1 PR = 1 機能単位
- **developブランチで開発**: feature/issue-N ブランチ → develop へPR → 区切りでmainへマージ

## フェーズ順序

必ずこの順番で進める。前フェーズの検証が通ってから次に進む。

1. **プロジェクト初期化** (#1)
2. **CPU実装** (#2-#8) → **nestest検証** (#9)
3. **カートリッジ読込** (#10)
4. **PPU実装** (#11-#14) → **SDL2描画統合** (#15)
5. **コントローラ入力** (#16)
6. **APU実装** (#17-#19) → **SDL2オーディオ統合** (#20)
7. **Mapper拡張** (#21-#23)
8. **Raspberry Pi最適化** (#24)

## テスト戦略

### ユニットテスト (`#[cfg(test)]`)

各モジュール内に `mod tests` を置く。

- **CPU**: 各命令が正しくレジスタ・フラグ・メモリを変更するか
  - 例: `LDA #$80` → A=0x80, N=1, Z=0
  - 例: `ADC` のキャリー・オーバーフロー境界値
  - アドレッシングモードごとにアドレス解決が正しいか
- **Bus**: アドレス範囲に応じて正しいデバイスにルーティングされるか
  - ミラー領域 ($0800-$1FFF → $0000-$07FF) の検証
- **PPU**: レジスタ読み書きの副作用 (ラッチ、VBlankフラグクリア等)
- **Cartridge**: iNESヘッダ解析、Mapper番号・ROM/RAMサイズ取得
- **Controller**: ストローブ→8回シリアル読取の動作

### 統合テスト (`tests/`)

- **nestest.nes**: CPU命令の網羅的検証 (公式ログと比較)
  - CPUフェーズ (#2-#8) 完了後に実施
  - ログ形式で各命令実行後のレジスタ状態を出力し、期待値と差分比較
- **画面出力テスト**: 既知のROMでスクリーンショット比較 (PPU実装後)

### テスト実行

```bash
cargo test              # 全テスト
cargo test cpu          # CPUモジュールのみ
cargo test -- --nocapture  # println!出力を表示
```

## ブランチ運用

```
main (安定版)
  └── develop (開発統合)
        └── feature/issue-1 (各Issue作業用)
```

- feature ブランチは develop から切る
- PR → develop へマージ
- フェーズの区切り (例: CPU完成) で develop → main へマージ

## コード規約

- `pub` は必要最小限。モジュール間のインターフェースのみ公開
- マジックナンバーは定数化 (`const`): アドレス範囲、レジスタビット位置など
- NES固有の値にはコメントで意味を付記 (例: `0x2002 // PPUSTATUS`)
- `unsafe` は使わない (SDL2のFFI境界を除く)
